# تحلیل Toornament و مقایسه با پروپوزال تو

> منبع‌ها: وب‌سایت رسمی Toornament، بلاگ، و مستندات توسعه‌دهندگان (لینک‌ها در انتهای سند). تاریخ بررسی: ژوئن ۲۰۲۶.

## ۱. Toornament چیست؟

یک پلتفرم SaaS بالغ و جهانی برای **مدیریت مسابقات esports/بازی‌های ویدیویی**. سازمان‌دهنده‌ها (از رویداد کوچک تا سرکیت بزرگ) کل چرخه‌ی مسابقه را روی آن می‌سازند: ثبت‌نام، ساختار/براکت، زمان‌بندی، ثبت نتیجه، رده‌بندی، رسانه/استریم، و انتشار عمومی. هم محصول آماده (داشبورد) دارد و هم **API کامل** و حتی **white-label**.

## ۲. مدل داده و ساختار (مهم‌ترین درسِ معماری)

Toornament یک سلسله‌مراتب دقیق دارد:

```
Tournament → Stage → Group → Round → Match → Game(set)
```
- **Discipline**: بازی (FIFA، CS و …)
- **Stage**: مرحله‌ی بزرگ مسابقه (مثلاً گروهی، سپس حذفی) — یک تورنومنت چند Stage دارد
- **Group**: زیرمجموعه‌ی هر Stage که هم‌زمان قابل‌اجراست
- **Round**: مجموعه مسابقاتی که هم‌زمان بازی می‌شوند
- **Match**: رویارویی دو طرف؛ **Game/Set**: زیربخش یک Match با نتیجه‌ی خودش
- **Participant / Registration / Custom Fields / Standings (rank در برابر position) / Circuit-Season**

👉 **در پروپوزال تو، «مسابقه» تقریباً تخت است** (یک مسابقه = یک‌سری بازی). برای پشتیبانی واقعی از فرمت‌های حرفه‌ای باید همین سلسله‌مراتب Stage/Group/Round را بپذیری.

## ۳. فرمت‌های ساختار (Bracket)

Toornament: **single elimination، double elimination، custom bracket، gauntlet، round-robin، league، Swiss، و FFA brackets**.

👉 پروپوزال تو فقط **Elimination و Round-Robin** را نام برده. این فاصله بزرگ است — موتور براکت سخت‌ترین و گران‌ترین بخش مهندسی است.

## ۴. امکانات کلیدی سازمان‌دهنده

- **ثبت‌نام**: تاریخ باز/بسته، سقف، پذیرش خودکار، seeding، و **Custom Fields** در سطح تیم و بازیکن (IGN، کشور، تاریخ تولد).
- **مدیریت**: راه‌اندازی/انتشار/کپی/آرشیو تورنومنت؛ **تعداد نامحدود ادمین**؛ گروه‌های دسترسی با سطوح و حقوق (RBAC).
- **انطباق GDPR** در پردازش داده‌ی بازیکن.
- **رده‌بندی**: rank (قابل‌اشتراک بین مساوی‌ها) و position (همیشه یکتا)، با tiebreaker.
- **رسانه/استریم**: ویدیو و استریم از طریق Media API.
- **پول‌سازی**: ثبت‌نام پولی؛ پلن Community حتی **۰٪ کارمزد Toornament** را به‌عنوان مزیت می‌فروشد.
- **Circuit/Season**: مدیریت چند تورنومنت پشت‌سرهم با لیدربرد سراسری.

## ۵. API و White-label

API کامل REST با این بخش‌ها: **Tournament، Structure، Match، Media، Platform، Circuit، Permission، Webhook**.
- احراز هویت: **API Key + OAuth2 (Client Credentials)** با scope.
- **Webhook** با تلاش مجدد تا ۲۴ ساعت.
- پلن‌های **Game Studios/Publishers**: Starter (API پایه)، Advanced (**اتوماسیون نتیجه‌ی on-demand، account linking، آمار پیشرفته**)، Platform (**white-label، جداسازی کامل داده و مالکیت کامل داده**).

> 🔑 **نکته‌ی طلایی:** اینکه «اتوماسیون نتیجه + account linking» فقط در پلن‌های **استودیوهای بازی** عرضه می‌شود، دقیقاً تأیید همان ایراد بازبینی است: یک شخص ثالث عادی **نمی‌تواند** نتیجه را خودکار از FIFA/Warzone بکشد؛ این کار نیازمند یکپارچگی در سطح ناشر بازی است.

## ۶. قیمت‌گذاری (۲۰۲۶)

پلن‌های سازمان‌دهنده:
- **Free**: تا ۳۲ شرکت‌کننده، همه‌ی فرمت‌ها.
- **Boost**: €۱۹ به‌ازای هر تورنومنت، تا ۱۲۸ نفر، Custom Fields + برندینگ.
- **Community** (جدید): €۱۹/ماه (سالانه) یا €۲۹/ماه — چند تورنومنت، ثبت‌نام پولی با ۰٪ کارمزد.
- **Circuit**: €۶۹/ماه (سالانه) یا €۸۹/ماه — مقیاس بزرگ + لیدربرد.
- **Arena** (جدید): ابزار انتشار/وب‌سایت اختصاصی + دسترسی API (قیمت سفارشی).

## ۷. مقایسه‌ی مستقیم با پروپوزال تو

| محور | Toornament | پروپوزال تو |
|------|-----------|-------------|
| مدل داده | سلسله‌مراتبی Stage/Group/Round/Match/Game | تقریباً تخت |
| فرمت براکت | ۸ نوع | ۲ نوع (Elim, Round-Robin) |
| seeding | دارد | ذکر نشده |
| check-in بازیکن | دارد | ذکر نشده |
| Custom Fields ثبت‌نام | تیم + بازیکن | کلی |
| RBAC ادمین | پخته، نامحدود | دارد (سطوح دسترسی) |
| تأیید نتیجه | دستی + (اتوماسیون فقط برای ناشر) | باید انسان‌در‌حلقه شود ✅ |
| پرداخت | بین‌المللی (Euro) | **زرین‌پال/ایران** ✅ (تمایز تو) |
| استریم | Media API | وعده داده (به فاز بعد رفت) ✅ |
| API عمومی | کامل + white-label | ندارد |
| بازار هدف | جهانی/انگلیسی | **ایران/فارسی** ✅ (تمایز تو) |

## ۸. تصمیم استراتژیک: بساز از صفر، یا روی Toornament بساز؟

سه گزینه:
- **A) ساخت از صفر** (پروپوزال فعلی): کامل‌ترین کنترل و مالکیت داده، ولی باید سخت‌ترین بخش (موتور براکت/ساختار) را از نو بسازی.
- **B) ساخت روی API تونومنت**: موتور براکت/ساختار/مسابقه‌ی اثبات‌شده را از API بگیری و فقط **تمایزهای خودت** (پرداخت ایرانی، UI فارسی، کامیونیتی محلی) را روش بسازی → سریع‌تر. **ولی:** وابستگی، هزینه‌ی یورویی، و مهم‌تر از همه **مشکل تحریم/پرداخت** (همان مشکل Stripe — پرداخت اشتراک یورویی از ایران سخت/ناممکن است و داده روی سرور اروپایی).
- **C) white-label (پلن Platform)**: مالکیت کامل داده، ولی مخصوص استودیوهای بازی و گران/سفارشی.

> 💡 **توصیه:** به‌خاطر تحریم و بازار ایران، گزینه‌ی B/C عملاً برایت دشوار است (همان دلیلی که Stripe را کنار گذاشتیم). پس **از صفر بساز، اما مدل داده و فهرست فرمت‌های Toornament را وام بگیر** و روی تمایز واقعی‌ات تمرکز کن: پرداخت ایرانی + فارسی + کامیونیتی محلی.

## ۹. اصلاحاتی که باید به پروپوزال/مدل تو اضافه شود

1. پذیرش سلسله‌مراتب **Stage → Group → Round → Match → Game** در مدل داده.
2. افزودن فرمت‌های براکت بیشتر (double elimination، Swiss، gauntlet، league، FFA).
3. افزودن یوزکیس‌های **seeding**، **check-in پیش از مسابقه**، و **Custom Fields ثبت‌نام (تیم/بازیکن)**.
4. مفهوم **Circuit/Season** و لیدربرد سراسری برای کامیونیتی.
5. تثبیت تأیید نتیجه به‌صورت **انسان‌در‌حلقه** (که در نسخه‌ی ۲ انجام شد) — همسو با واقعیت Toornament.
6. تعریف صریح **تمایز رقابتی**: چرا یک سازمان‌دهنده‌ی ایرانی به‌جای Toornament از محصول تو استفاده کند؟ (پرداخت ریالی، فارسی، پشتیبانی محلی).
